# v0.4 Hearth — Roadmap

- **Date**: 2026-04-18
- **Status**: Roadmap approved. L3a Recipe Library spec next.
- **Authors**: Hana (product), Nell (architect)

## Purpose

Bring meal planning, recipes, and grocery shopping into Manor as a single coherent area, shipped in small weekly slices so value lands continuously. Parallel structure to `2026-04-16-gap-closure-roadmap.md` — this roadmap sequences v0.4 sub-landmarks into discrete, independently shippable chunks.

---

## Scope decisions (locked during brainstorm)

| Area | Decision |
|---|---|
| **Pantry** | Lightweight "staples" exclusion list only. No inventory. No expiry. |
| **URL import** | Hybrid: schema.org JSON-LD first, LLM fallback. Parse-path badge shown to user. |
| **Meal plan granularity** | One slot per day (implicit "dinner"). |
| **Calendar integration** | Internal only. Today + TimeBlocks reflect tonight's meal. No CalDAV write for v0.4. |
| **Shopping list lifecycle** | One active list. Regenerate overwrites (no tick-state preservation). |
| **Ingredient dedupe** | Itemised with recipe provenance. No quantity math across recipes. |
| **Meal ideas reshuffle** | Default: library rotation. "Something new" pivot: LLM (local → remote tier). |

---

## Sub-landmarks (dependency-ordered)

Four landmarks. Each ships as its own spec + implementation plan. Pattern matches v0.1–v0.3 releases.

### L3a — Recipe library — 🎯 **NEXT**

Ships alone, genuinely useful as "a place for my recipes."

- `recipe` + `recipe_ingredient` schema
- CRUD UI (list, detail, edit)
- Hybrid URL import (schema.org JSON-LD → LLM fallback)
- Parse-path badge ("parsed from structured data" vs "AI-extracted — double-check")
- Recipe tags (reuses shared `tag` + `tag_link` from foundation)
- Recipe images (reuses shared `attachment` from foundation)
- Hearth nav entry point (new top-level tab or section)

**Depends on**: foundation tables (shipped in v0.1 Completion Phase A).

### L3b — Meal plan + staples — depends on L3a

- `meal_plan` + `meal_plan_entry` schema
- Weekly grid UI — assign recipes to days
- Today band reflection ("Tonight: miso aubergine")
- TimeBlocks view reflection
- `staple_item` schema — a list of "things I usually have" that shopping generator excludes by default
- Staples management UI

**Depends on**: L3a.

### L3c — Shopping list — depends on L3a + L3b

- `shopping_list` + `shopping_list_item` schema
- Generator: "give me the next 7 days of the meal plan, minus staples" — itemised with recipe provenance
- One active list at a time; regenerate overwrites
- Tick-off UI
- Manual additions (non-recipe items: bin bags, etc.)

**Depends on**: L3a + L3b.

### L3d — Meal ideas reshuffle — depends on L3a + L3b

- Meal ideas panel (Hearth home surface + Today band?)
- Default source: rotate from recipe library (recipes not cooked recently)
- "Something new" pivot: Ollama local → remote tier per existing Remote LLM plumbing
- AI-suggested recipe → one-tap save into library
- Reshuffle button cadence + rate limiting

**Depends on**: L3a + L3b, existing Ollama + Remote LLM (shipped in Landmark 2).

## Out of scope for v0.4 (explicit)

Listing so we don't drift:

- Pantry inventory tracking. Expiry dates. "Used half a kilo of onions" bookkeeping.
- Nutritional information. Calorie counting.
- Shared meal plans across household members (persons exist but meal assignees are not MVP).
- CalDAV push of meal plan entries. Deferred to v0.4.1 earliest.
- Recipe scaling ("make this for 6 instead of 4"). Deferred.
- Ingredient quantity math across recipes in shopping list. Deferred.
- Multi-week meal planning (templates, "repeat last week"). Deferred.
- Recipe sharing / export / import between Manor installs. Deferred.

---

## Sequencing rationale

Why these boundaries:

- **L3a before everything** — a recipe library is useful even without meal planning. Validates URL import parser in isolation before higher layers depend on it.
- **L3b before L3c** — shopping list has nothing to generate from until meal plan exists.
- **L3d last** — meal ideas reshuffle benefits from seeing a populated recipe library and real meal-plan history before we invest in LLM integration.
