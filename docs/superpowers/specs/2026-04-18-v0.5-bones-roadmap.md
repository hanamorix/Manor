# v0.5 Bones — Roadmap

- **Date**: 2026-04-18
- **Status**: Roadmap approved. L4a Asset Registry spec next.
- **Authors**: Hana (product), Nell (architect)

## Purpose

Bring house-operations into Manor as a single coherent area: what do I own, when does it need service, what did it cost, and when something breaks — what do I do? Shipped as a sequence of independently-valuable sub-landmarks, following the v0.4 Hearth pattern.

Parallel structure to `2026-04-18-v0.4-hearth-roadmap.md` — this roadmap sequences v0.5 sub-landmarks into discrete, independently-shippable chunks.

---

## Scope decisions (locked during brainstorm)

| Area | Decision |
|---|---|
| **Tab name** | **Bones** (matches Hearth/Ledger/Rhythm naming pattern — "the skeletal frame of the house: pipes, walls, appliances"). |
| **MVP daily question** | "When's my boiler service due?" — schedules + due-date computation is the heart. Registry is the substrate. |
| **Right-to-repair** | First-class landmark (L4d) — symptom-search pulls from ifixit.com + repairclinic.com + similar, LLM synthesises top hits into a readable summary, saved as troubleshooting notes per asset. |
| **PDF manual extraction** | In scope (L4e) — LLM reads a PDF attached to an asset and proposes maintenance_schedule entries. |
| **Ledger integration** | `maintenance_event.transaction_id` links service costs to Ledger transactions. Cost history rolls up per asset. |
| **Asset attachments** | Reuses L3a's attachment infrastructure. Hero image + N PDFs per asset (manuals, warranties, invoices). |
| **Category taxonomy** | Minimal MVP: `appliance | vehicle | fixture | other`. No sub-categories, no industry taxonomy. User can add free-form tags via the shared `tag` table. |

## Sub-landmarks (dependency-ordered)

Five landmarks. Each ships as its own spec + implementation plan.

### L4a — Asset registry — 🎯 **NEXT**

Ships alone, genuinely useful as "a list of stuff I own with photos, manuals, and purchase info."

- `asset` schema + CRUD
- Category enum (appliance / vehicle / fixture / other)
- Fields: name, category, make, model, serial, purchase date, notes, hero image
- PDF + image attachments (reuses L3a's `attachment` infra; `entity_type='asset'`)
- Tags via shared `tag_link`
- New **Bones** top-level nav tab (Lucide icon — probably `Wrench` or `Hammer`)

**Depends on**: foundation tables (shipped v0.1 Completion).

### L4b — Maintenance schedules + due dates — depends on L4a

- `maintenance_schedule` schema: asset_id, task description, interval (months/years/usage-units), last_done_date, next_due_date
- Interval kinds: time-based (`every 12 months`), usage-based (`every 10000km` for vehicles), either-first (`12 months or 10000km, whichever comes first`)
- Due-date computation per schedule
- Bones overview view: "Due soon / Overdue / Upcoming" banded list
- Today-band reflection when a schedule is overdue
- Rhythm integration: overdue maintenance appears as a high-priority chore item

**Depends on**: L4a.

### L4c — Maintenance events + Ledger link — depends on L4a + L4b

- `maintenance_event` schema: schedule_id (optional — one-off work has no schedule), asset_id, completed_date, notes, cost_pence, transaction_id (optional link to Ledger)
- "Mark schedule done" flow — captures event + optional cost + bumps the schedule's `last_done_date` + recomputes `next_due_date`
- Per-asset history log
- Cost roll-up per asset + per category ("How much did I spend on the house last year?")
- Ledger linkage: if the user has recent matching transactions, suggest linking one on event creation

**Depends on**: L4a + L4b.

### L4d — Right-to-repair lookup — depends on L4a

- Per-asset "Something's wrong" search interface
- User types a symptom ("dishwasher won't drain," "fridge making grinding noise")
- Manor fetches from curated sources (ifixit.com search results, repairclinic.com model-search, possibly YouTube search result pages)
- LLM synthesises the top 2–3 hits into a readable summary with source URLs preserved
- Results saved as a `repair_note` (or reuse the shared `note` table with `entity_type='asset'`) so history is searchable later
- Tier routing reuses Landmark 2 plumbing (local Ollama first → remote fallback)

**Depends on**: L4a, existing Ollama + Remote LLM (Landmark 2), existing reqwest fetch patterns from L3a.

### L4e — PDF manual extraction — depends on L4a + L4b

- User attaches a PDF manual (boiler, washing machine, car handbook) to an asset
- "Extract maintenance schedule from this PDF" action on the attachment
- LLM parses the PDF text (via `pdf-extract` crate or similar local library → plain text) looking for scheduled-maintenance sections
- Proposes a set of `maintenance_schedule` entries for the user to approve/edit/reject (reuses Manor's existing `proposal` table pattern per overall design §4.5)
- User approves selected proposals → schedules land in the database

**Depends on**: L4a, L4b, existing Ollama + Remote LLM, L3a's LLM extract pattern.

## Out of scope for v0.5 (explicit)

Pinned to prevent drift:

- Warranty expiry tracking + alerts. Could be v0.5.1 on top of existing attachment metadata.
- Vehicle-specific features beyond the shared schema (MOT dates, insurance renewal, fuel logs — those are really Ledger territory).
- Multi-property support (second home, rental). Single-household only for MVP.
- Room/location taxonomy beyond free-form tags.
- Depreciation calculation.
- Smart-home device integration (HomeKit, Matter, etc).
- Barcode / QR lookup for fast asset entry.
- Service-provider directory ("plumbers near me") — out of scope forever, not Manor's job.
- Shared maintenance calendars across households.
- Real-time parts-availability checking on repair lookups.
- Automatic proposal-generation from transaction history ("you paid £180 to British Gas — log a boiler service?"). Interesting but v0.5.1 at earliest.

## Sequencing rationale

Why these boundaries:

- **L4a before everything** — asset is the root entity every other landmark depends on.
- **L4b next** — answers the daily "when's my boiler due" question. Ships schedule machinery without cost-tracking complexity.
- **L4c closes the money loop** — cost attribution + Ledger integration. Requires L4b's schedule rows to target.
- **L4d could ship any time after L4a** — it attaches repair notes to assets. Sequenced fourth because schedules + events + costs are the daily-driver features; repair lookup is event-driven ("when something breaks"), not daily.
- **L4e last** — PDF extraction writes INTO `maintenance_schedule` rows that L4b defines. Landing it before L4b is backwards. It's also the riskiest LLM surface — better to let L4d season the LLM-synthesis patterns first.

### Shared architecture between L4d and L4e

Both are "LLM reads text, produces structured output, user approves." They can share:

- A common `llm_extract` module pattern for prompt + retry + JSON-block parsing (already well-trodden in L3a).
- The existing `proposal` table (overall design §4.5) for user-approval workflows — L4e writes proposals directly; L4d could optionally wrap repair summaries as proposals if we want undo-ability.

Neither needs to share code yet; flag during L4e implementation if duplication becomes obvious.
