## 1. Phase A — Foundation tables

Five SQLite tables + DAL modules + migrations. No UI except where called out.

All tables follow the **universal row shape** from design §4.1:
```sql
id          INTEGER PRIMARY KEY,   -- autoincrement; UUIDv7 deferred to v1.0 sync work
created_at  INTEGER NOT NULL DEFAULT (unixepoch()),
updated_at  INTEGER NOT NULL DEFAULT (unixepoch()),
deleted_at  INTEGER                 -- soft-delete
```

**Deviation from overall design**: we keep INTEGER PKs (matching existing tables) rather than TEXT/UUIDv7. UUIDv7 migration is a v1.0 sync concern; introducing it here would churn six existing phases. Document the deviation in the overall design.

### 1.1 `setting(key, value)` — first

Simplest. Used to store non-secret preferences (trash auto-empty duration, weather location, default LLM model, onboarding completion flag).

```sql
CREATE TABLE setting (
    key        TEXT PRIMARY KEY,
    value      TEXT NOT NULL,
    updated_at INTEGER NOT NULL DEFAULT (unixepoch())
);
```

DAL: `get(key) -> Option<String>`, `get_with_default(key, default)`, `set(key, value)`, `delete(key)`, `list_prefixed(prefix)`.

JSON-typed helper: `get_json<T: DeserializeOwned>(key) -> Option<T>` for structured values.

### 1.2 `person` + `household`

```sql
CREATE TABLE person (
    id           INTEGER PRIMARY KEY,
    name         TEXT NOT NULL,
    kind         TEXT NOT NULL CHECK (kind IN ('owner','member','contact','provider','vendor')),
    email        TEXT,
    phone        TEXT,
    note         TEXT,
    created_at   INTEGER NOT NULL DEFAULT (unixepoch()),
    updated_at   INTEGER NOT NULL DEFAULT (unixepoch()),
    deleted_at   INTEGER
);

CREATE TABLE household (
    id                 INTEGER PRIMARY KEY CHECK (id = 1),  -- singleton
    owner_person_id    INTEGER REFERENCES person(id),
    working_hours_json TEXT NOT NULL DEFAULT '{"mon":[9,17],"tue":[9,17],"wed":[9,17],"thu":[9,17],"fri":[9,17],"sat":[],"sun":[]}',
    dnd_windows_json   TEXT NOT NULL DEFAULT '[]',
    created_at         INTEGER NOT NULL DEFAULT (unixepoch()),
    updated_at         INTEGER NOT NULL DEFAULT (unixepoch())
);

INSERT INTO household (id) VALUES (1);
```

**Integration with Phase 4 Rhythm**: chores currently use an `add_person` command that appears to create person rows elsewhere. Before adding this schema, audit `crates/app/src/rhythm/commands.rs::add_person` and migrate to the new `person` table. One-commit data migration inside the V8 migration.

DAL: standard CRUD on `person`. `household::get() -> Household` (always returns id=1).

### 1.3 `tag` + `tag_link`

```sql
CREATE TABLE tag (
    id         INTEGER PRIMARY KEY,
    name       TEXT NOT NULL UNIQUE COLLATE NOCASE,
    color      TEXT NOT NULL DEFAULT '#888',
    created_at INTEGER NOT NULL DEFAULT (unixepoch())
);

CREATE TABLE tag_link (
    id          INTEGER PRIMARY KEY,
    tag_id      INTEGER NOT NULL REFERENCES tag(id) ON DELETE CASCADE,
    entity_type TEXT    NOT NULL,   -- 'event' | 'task' | 'transaction' | 'attachment' | 'note' | 'person'
    entity_id   INTEGER NOT NULL,
    created_at  INTEGER NOT NULL DEFAULT (unixepoch()),
    UNIQUE(tag_id, entity_type, entity_id)
);

CREATE INDEX idx_tag_link_entity ON tag_link(entity_type, entity_id);
```

DAL: `upsert_tag(name, color)`, `link(tag_id, entity_type, entity_id)`, `unlink(...)`, `tags_for(entity_type, entity_id) -> Vec<Tag>`, `entities_with_tag(tag_id) -> Vec<(String, i64)>`.

No tag-UI in this phase — just the DAL. Tags surface in v0.4+ UI.

### 1.4 `note`

```sql
CREATE TABLE note (
    id          INTEGER PRIMARY KEY,
    body_md     TEXT NOT NULL,
    entity_type TEXT,          -- optional parent; NULL = orphan note
    entity_id   INTEGER,
    created_at  INTEGER NOT NULL DEFAULT (unixepoch()),
    updated_at  INTEGER NOT NULL DEFAULT (unixepoch()),
    deleted_at  INTEGER
);

CREATE INDEX idx_note_entity ON note(entity_type, entity_id) WHERE deleted_at IS NULL;
```

DAL: CRUD + `list_for(entity_type, entity_id)`. No rendering component yet — exposed for future skills.

### 1.5 `attachment` — file-backed

```sql
CREATE TABLE attachment (
    id              INTEGER PRIMARY KEY,
    uuid            TEXT    NOT NULL UNIQUE,              -- filesystem name
    original_name   TEXT    NOT NULL,
    mime_type       TEXT    NOT NULL,
    size_bytes      INTEGER NOT NULL,
    sha256          TEXT    NOT NULL,                     -- dedup + integrity
    entity_type     TEXT,                                  -- optional parent
    entity_id       INTEGER,
    created_at      INTEGER NOT NULL DEFAULT (unixepoch()),
    deleted_at      INTEGER
);

CREATE INDEX idx_attachment_entity ON attachment(entity_type, entity_id) WHERE deleted_at IS NULL;
CREATE INDEX idx_attachment_sha    ON attachment(sha256);
```

Bytes live at `~/Library/Application Support/Manor/attachments/<uuid>` (actual path resolved via `app_data_dir()`).

DAL: `store(bytes, original_name, mime, entity_type, entity_id) -> Attachment` (computes sha256, writes file), `get_bytes(id) -> Vec<u8>`, `delete(id)` (soft-delete row + garbage-collect file on next permanent-delete sweep), `list_for(entity_type, entity_id)`.

**Dedup**: if `sha256` already exists on a non-deleted row, reuse the existing file and just insert a new row pointing at the same `uuid`. Saves disk on duplicate PDFs.

## 2. Phase B — Deletion + safety

Soft-delete already exists on most tables. This phase adds the user-facing surfaces and the auto-cleanup job.

### 2.1 Trash view

- New route / tab in Settings → Data & Backup.
- Shows every row across every table with `deleted_at IS NOT NULL`.
- Groups by entity type; each row shows title, entity type, deleted-at timestamp, days-until-auto-empty.
- Actions: **Restore** (clear `deleted_at`) / **Delete permanently** (hard DELETE + file cleanup for attachments).
- One bulk "Empty trash now" button, double-confirmed.

DAL: a thin aggregator that unions `SELECT ... FROM <table> WHERE deleted_at IS NOT NULL` across the known tables. Hard-coded list of table names — no dynamic introspection.

### 2.2 Auto-empty job

Run on app open (same `spawn_blocking` slot as the ledger auto-insert already uses):
```rust
DELETE FROM <table> WHERE deleted_at IS NOT NULL
  AND deleted_at < (unixepoch() - auto_empty_days * 86400);
```
For `attachment`, also delete the file at `attachments/<uuid>` before removing the row. Wrap in a single transaction per table.

`auto_empty_days` is stored in `setting` (keys: `trash.auto_empty_days`, default `30`). Valid values: `7`, `30`, `90`, `never` (never = don't run).

### 2.3 Panic button

Settings → Data & Backup → **"Erase everything"**.

- Requires typed confirmation: user must type the word `DELETE`.
- Offers to export a snapshot first (routes through Phase B.4 backup flow).
- On confirm: closes DB connection → deletes `manor.db` + `attachments/` → restarts the app into the first-launch wizard.

### 2.4 Weekly encrypted snapshot (`.lifebackup`)

Writes `<chosen_dir>/manor-YYYY-WW.lifebackup` once per week on a configurable day (default Sunday 02:00 local time).

Format: `tar.gz` of `manor.db` + `attachments/` + `config.toml`, piped through `age` encryption. Passphrase derived from a user-chosen phrase held in Keychain under `com.hanamorix.manor.backup.passphrase`.

Restore: drag-and-drop a `.lifebackup` onto the first-launch wizard step 1, or from Settings → Data & Backup → "Restore from backup".

Dependencies to add:
- `age = "0.10"` (Rust `age` crate for encryption)
- `tar = "0.4"` + `flate2 = "1"` for the archive

Uses `launchd` for running when app is closed (`~/Library/LaunchAgents/com.hanamorix.manor.snapshot.plist`). Registered from the first-launch wizard step, removed by the panic button.

## 3. Phase C — Embeddings / `sqlite-vec`

The hardest Phase A dependency — adds a loadable SQLite extension.

### 3.1 Extension loading

- `sqlite-vec` ships as a dylib. We vendor a pinned copy at `crates/core/assets/vec0.dylib` (architecture-specific; build.rs fetches per-target).
- `rusqlite` loads it at connection open via `conn.load_extension_enable()` + `conn.load_extension("vec0.dylib", None)`.
- On load failure: log warning, keep app running, embeddings features silently disabled (UI shows an info banner in Settings → AI).

### 3.2 `embedding` table

```sql
CREATE VIRTUAL TABLE embedding USING vec0(
    entity_type TEXT NOT NULL,
    entity_id   INTEGER NOT NULL,
    model       TEXT NOT NULL,
    updated_at  INTEGER NOT NULL,
    vector      float[768]           -- nomic-embed-text default
);
```

Separate index-style table for `(entity_type, entity_id)` → embedding_id lookup, since `vec0` virtual tables don't support secondary indexes:

```sql
CREATE TABLE embedding_ref (
    entity_type  TEXT    NOT NULL,
    entity_id    INTEGER NOT NULL,
    embedding_id INTEGER NOT NULL,
    model        TEXT    NOT NULL,
    updated_at   INTEGER NOT NULL,
    PRIMARY KEY (entity_type, entity_id, model)
);
```

### 3.3 Embed-on-write pipeline

Not a live hook on every DAL `insert`. Instead: a background job that runs every 5 minutes (in-app, when open) via the existing scheduler-like pattern used for calendar sync:

1. Walk known entity types: `note`, `attachment` (where mime is text-like), `ledger_transaction.description`.
2. For each, find rows where `embedding_ref.updated_at < <entity>.updated_at` OR no ref exists.
3. For each stale row, call `OllamaClient::embed(text) -> Vec<f32>` (new method on the existing client using `/api/embeddings`).
4. Upsert into `embedding` + `embedding_ref`.
5. Cap at 200 rows per tick; resume next tick.

Default model: `nomic-embed-text` via Ollama. Configurable via `setting` key `embeddings.model`.

### 3.4 Query API

`fn search_similar(text: &str, entity_types: &[&str], limit: usize) -> Vec<(String, i64, f32)>`

Used by future skills. No UI in this phase — just the DAL.

## 4. Phase D — Today polish

### 4.1 Weather strip

`wttr.in` free API: `https://wttr.in/<location>?format=j1` returns JSON with current condition + today's forecast.

Location: stored in `setting` key `today.weather_location` (default: IP-derived — wttr.in does this if you omit location). Editable in Settings → Household.

Cache: TTL 30 minutes in memory + spill to `setting` key `today.weather_cache` (JSON blob with timestamp). On offline: use cached blob if <24h old, else render a dash.

UI addition in `HeaderCard.tsx`: below the date line, one row showing `<emoji> <temp>°C, <condition>`. No emoji → no weather section; graceful degradation.

### 4.2 First-launch wizard

Tauri window opens the wizard instead of the main app when `setting.onboarding_completed != '1'`. Four steps:

1. **Where does your life live?** — default `~/Library/Application Support/Manor/`. "Change" button opens a native folder picker; sets `setting.data_dir`. If user picks a custom path, the `manor.db` and `attachments/` move there on confirm. Restore-from-backup link surfaces here.
2. **Meet your brain** — detect Ollama at `localhost:11434`. If present: "✓ Ollama ready — using `qwen2.5:7b-instruct`". If absent: install instructions + "skip for now" (degrades gracefully; most skills still work without AI).
3. **Your calendar** — reuses existing `AddAccountForm` component (already built). Skip-able.
4. **Show me what I can do** — three sample-data buttons: *"Plan my week (sample)"*, *"Add a chore (sample)"*, *"Write a shopping list (sample)"*. Clicking each inserts labelled sample rows (`is_sample = TRUE`, separate column added to affected tables — event, task, chore). A persistent banner at top of Today until user clicks "Delete sample data".

Final step: `setting.set('onboarding_completed', '1')` + navigate into Today.

Resumable: if the user quits mid-wizard, it reopens on the last unvisited step.

## 5. Phase E — Settings tabs + housekeeping

### 5.1 Settings tabs

Replace single "Calendar" modal with five tabs in the existing SettingsModal:

**Tab 1: Data & Backup**
- Current data directory (editable via folder picker).
- Last snapshot timestamp + "Back up now".
- Trash subview (Phase B).
- Panic button (Phase B.3).
- Trash auto-empty duration selector (7 / 30 / 90 / never).
- Snapshot schedule picker (weekly day-of-week + time).

**Tab 2: AI**
- Ollama status (reachable / model loaded / model name + version).
- Default model selector (reads Ollama's `/api/tags`).
- Embeddings status + "Rebuild embeddings" button.
- Placeholder for Landmark 2: "Remote providers will appear here."

**Tab 3: Calendar & Accounts** (existing — unchanged)

**Tab 4: Household**

## 6. Non-goals (for v0.1 Completion)

- No remote LLM, no provider abstraction, no keys UI — Landmark 2.
- No sync between devices — v1.0 Companion.
- No iCloud Drive / Dropbox-specific tooling — user points `data_dir` at any folder.
- No UUIDv7 migration — keep INTEGER PKs, document the deviation.
- No new skills (no meals, no home assets).
- No cross-table reporting beyond what Trash view needs.
- No rich tag UI; tags are DAL-only this phase.

## 8. Open questions

1. **Sample-data flag**: adding `is_sample` columns to existing tables churns the migration history. Alternative: a single `sample_data_active` setting key + a known tag (`#sample`) applied via `tag_link`. Leaning toward the tag-based approach — cleaner.
2. **`sqlite-vec` vendoring**: fetch at build time, or commit binary? Target-specific binary is cross-platform-hostile but deterministic. Proposal: fetch in build.rs per target, document in CI.
3. **Backup passphrase recovery**: if user loses it, backups are unrecoverable. Do we show a "last 4 chars of your key" hint on the restore screen? Proposal: yes, stored in `setting.backup.passphrase_hint`.
4. **Weather without internet on first launch**: default to no weather until first successful fetch, vs. render a "connect to see weather" string. Proposal: show nothing — §6.5 calm-over-clever.
5. **Household model for single-person use**: does the solo user ever want to set `owner_person_id = NULL`? Proposal: no, auto-create "You" person on first launch.
6. **Wizard restart logic**: if user nukes onboarding flag manually, do we preserve the rest of their data? Proposal: yes — wizard detects existing DB content and says "picking up where you left off" instead of sample-data step.
