# Life Assistant v0.1 Heartbeat — Phase 1: Foundation Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Establish a buildable, runnable Tauri 2 + Rust monorepo that opens a window, shows "Life Assistant", and successfully round-trips a `ping` IPC command from the React shell to the Rust core. Green CI, green `cargo test`, clean `pnpm tsc`.

**Architecture:** Cargo workspace with `crates/core` (pure Rust library) and `crates/app` (Tauri-commands glue crate). React + TypeScript shell lives at `apps/desktop/` with its own pnpm-managed workspace. Tauri 2 shell links `crates/app` as a dependency; no sidecar, no daemon. GitHub Actions CI runs `cargo fmt`, `cargo clippy`, `cargo test`, and `pnpm tsc` on macOS runners.

**Tech Stack:** Rust stable 1.80+, Cargo workspace (resolver 2), Tauri 2.x, React 18 + TypeScript 5.5 + Vite 5, pnpm 9 workspace, Node 20 LTS, GitHub Actions.

---

## Plan sequencing note

Life Assistant v0.1 Heartbeat ships in **eight phases**. This document covers **Phase 1 only**. Each subsequent phase gets its own detailed plan written at the start of that phase. This matches the spec's "each release gets its own implementation plan" pattern.

| # | Phase | What it delivers |
|---|---|---|
| **1** | **Foundation** *(this plan)* | Workspace skeleton, Tauri boot, CI green, ping IPC works |
| 2 | Core data layer | SQLite, refinery migrations, ontology types, keychain |
| 3 | Models layer | Provider trait, Ollama provider, tier-based router |
| 4 | Calendar skill | CalDAV read client, event sync scheduler |
| 5 | Proposal + first AI tool | Proposal table, context recipes, `summarise_today` |
| 6 | React shell UI | Today view (4 bands), Assistant panel, Tailwind + shadcn/ui |
| 7 | First-launch wizard + Settings | 4-step wizard, five-section Settings |
| 8 | Packaging + signing | Ad-hoc signed DMG, release workflow |

```tsx
import { useEffect, useState } from "react";
import { ping, type PingResponse } from "./lib/ipc";

type State =
  | { kind: "loading" }
  | { kind: "ok"; resp: PingResponse }
  | { kind: "error"; message: string };

export default function App() {
  const [state, setState] = useState<State>({ kind: "loading" });

  useEffect(() => {
    ping()
      .then((resp) => setState({ kind: "ok", resp }))
      .catch((e: unknown) =>
        setState({ kind: "error", message: String(e) }),
      );
  }, []);

  return (
    <main style={{ padding: "2rem" }}>
      <h1>Life Assistant</h1>
      {state.kind === "loading" && <p>Contacting core…</p>}
      {state.kind === "ok" && (
        <p>
          Core says <strong>{state.resp.message}</strong> (core version{" "}
          {state.resp.core_version}).
        </p>
      )}
      {state.kind === "error" && (
        <p style={{ color: "#b91c1c" }}>Error: {state.message}</p>
      )}
    </main>
  );
}
```

## Plan deviations made in Task 4

**Step 6 — Icons (RGBA format required):**
The plan said to use renamed PNG copies as placeholders and noted Tauri might not validate formats during `cargo check`. In practice, Tauri's `generate_context!()` proc-macro validates icons at compile time and rejects non-RGBA PNGs with `icon … is not RGBA`. The initial placeholder (RGB PNG) caused a compile error. Fixed by regenerating all placeholder icons as RGBA PNGs using a Python script. The icons remain placeholders at 128×128 px — real icons still need to be generated via `tauri icon` after Task 9 installs pnpm.

**Step 6 — Icon download URL:**
The `curl` URL in the plan (`raw.githubusercontent.com/tauri-apps/tauri/dev/crates/tauri-cli/src/helpers/template/tauri/icons/icon.png`) returned 404. Fell back to generating a minimal valid RGBA PNG locally via Python's `struct` + `zlib` — no external tools required.

## Plan deviations made in Task 6

**Step 3 — `ping` function location:**
The plan put `#[tauri::command] fn ping` directly in `crates/app/src/lib.rs` alongside `tauri::generate_handler![ping]`. On Tauri 2.10.3 this triggers `error[E0255]: the name __cmd__ping is defined multiple times` — `#[tauri::command]` and `generate_handler!` both define `__cmd__ping` in the same module scope. Fixed by wrapping the command in a `mod commands { }` submodule and referring to it as `commands::ping` in `generate_handler!`. A `pub use commands::ping;` re-export keeps the test and any future callers unchanged. All tests pass 2/2, clippy clean, desktop shell still builds.
